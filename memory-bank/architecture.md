# Архитектура TON Wallet (Self-Custodial, Testnet)

## 1. Основной стек технологий
*   **Фронтенд:** Vue 3 (Composition API) с Pinia для управления состоянием и Vue Router для навигации.
*   **Стилизация:** Tailwind CSS для быстрого и консистентного UI.
*   **Иконки:** Heroicons для всех иконок в приложении.
*   **Типизация:** TypeScript для всей кодовой базы.
*   **Сборка:** Vite для сервера разработки и финальной сборки, с плагином `vite-plugin-node-polyfills` для совместимости Node.js модулей в браузере.

## 2. Поток данных и управление состоянием
*   **Pinia (`wallet-manager.store.ts` & `active-wallet.store.ts`):** Ядро управления состоянием кошелька разделено на два модуля.
    *   **Wallet Manager Store**: Управляет списком сохраненных кошельков (`savedWallets`) и текущим выбранным индексом. Синхронизируется с `localStorage` автоматически через `watch`.
    *   **Active Wallet Store**: Управляет состоянием **текущей сессии** (когда кошелек разблокирован).
        *   `isUnlocked`: Булево значение, указывающее, разблокированы ли ключи.
        *   `address`, `balance`, `transactions`, `whitelist`: Данные активного кошелька.
        *   `isLoadingData`, `dataError`: Флаги загрузки и обработки сетевых ошибок.
        *   `keyPair`: Дешифрованная пара ключей (в оперативной памяти).
*   **`localStorage`:** Используется для постоянного хранения:
    *   `saved_wallets`: JSON-строка массива `StoredWallet[]`.
    *   `current_wallet_index`: Индекс последнего активного кошелька.
    *   `wallet_whitelist_<address>`: Отдельный белый список для каждого кошелька.
*   **UI-компоненты:** Все компоненты Vue реактивно взаимодействуют с Pinia Store для отображения данных и вызова действий.

## 3. Ключевые сервисы (бизнес-логика и внешние взаимодействия)
*   **`crypto.service.ts`:**
    *   **Генерация/Валидация мнемоники:** Использует `@ton/crypto`.
    *   **Шифрование/Дешифрование:** Использует `crypto-js` (PBKDF2, AES-256).
*   **`ton.service.ts`:**
    *   **TON Client:** Инициализирует `TonClient` (`@ton/ton`) с использованием TonCenter Testnet API.
    *   **Запросы к сети:** `getBalance` и `getTransactions`.
    *   **Оценка комиссии:** `estimateFee` делает запрос к `estimateExternalMessageFee` для динамического расчета точной стоимости транзакции перед ее отправкой.
    *   **Отправка транзакций:** `sendTransaction` формирует, подписывает и отправляет BOC. `waitForTransaction` опрашивает сеть.
*   **`address-guard.service.ts`:**
    *   **Защита от Spoofing-атак:** Анализирует введенный адрес получателя на совпадения с `whitelist` и "похожесть" с историей транзакций. Возвращает `Evaluation` с `RiskLevel`.

## 4. Пользовательский интерфейс (Views & Components)
*   **`App.vue`:** Корневой компонент, содержит `RouterView`.
*   **`Auth.vue`:** Управление кошельками (выбор, создание, импорт, удаление, разблокировка).
*   **`Home.vue`:** Главный экран (баланс, кнопки Send/Receive, история активности, ошибки загрузки `dataError`).
*   **`WalletSend.vue`:** Экран отправки TON.
    *   Форма с динамическим расчетом комиссии (`estimatedFee`) через вызов `updateFeeEstimate` при вводе адреса/суммы.
    *   Отображение предупреждений `RiskLevel` от Spoofing Protection.
    *   Асинхронная отправка транзакции с индикаторами.
*   **`WalletReceive.vue`:** Экран получения TON (QR-код и кнопка копирования).

## 7. Компромиссы и принятые допущения

### 7.1. Проблема комиссий (Fees & Estimation)
**Решено (Частично):** В ранней версии использовалась фиксированная комиссия `0.01 TON`. Сейчас реализован динамический расчет комиссии через метод `tonService.estimateFee`, который вызывается при вводе данных на экране `WalletSend.vue`. 
**Остаточный риск:** При очень резких скачках нагрузки в сети или при сложной структуре смарт-контракта получателя (что в данном MVP не учитывается), итоговая комиссия всё равно может незначительно отличаться от оценки. Если API оценки `estimateExternalMessageFee` падает, система использует fallback-значение `0.01 TON`.

### 7.2. Проблема «Длинной истории» (History & Search)
**Проблема:** Блокчейн-узлы (RPC) отдают историю транзакций порциями (пагинация). Поскольку у нас нет своего бэкенда, мы не можем проиндексировать все транзакции пользователя в базе данных.
**Принятый компромисс:** Функция "Поиск по истории" работает только для **ограниченного числа загруженных транзакций** (50 последних транзакций) из TonCenter API. В UI нет механизмов пагинации или "бесконечного скролла" для загрузки более старых транзакций.
**Риски и будущие улучшения:** Если у кошелька много транзакций, поиск старых не сработает, пока они не будут загружены. Для production-версии потребуется либо реализация загрузки по требованию (`offset`, `before_lt`), либо использование специализированных индексаторов/бэкенда. 

### 7.3. Упрощенный парсинг транзакций:
**Проблема:** В `ton.service.ts` используется примитивная логика определения типа транзакции (входящая/исходящая) и контрагента.
**Принятый компромисс:** Данный функционал не был описан в исходных требованиях (ТЗ), поэтому в текущей версии реализовано базовое решение без избыточных усложнений».  
**Риски и будущие улучшения:** При взаимодействии со сложными смарт-контрактами (где может быть несколько сообщений) эта логика будет выдавать некорректные данные.

## 8. Механизмы безопасности (Security)

### 8.1. Локальное хранение (Local Storage & Encryption)
* Мнемоники никогда не покидают устройство пользователя и хранятся в `localStorage` исключительно в зашифрованном виде.
* Для шифрования применяется алгоритм AES-256 (через `crypto-js`) с предварительным усилением ключа с помощью PBKDF2.

### 8.2. Address Guard (Защита от Spoofing-атак)
В приложении реализован сервис `address-guard.service.ts` для проактивной защиты пользователя от атак "нулевыми переводами" (Address Spoofing/Poisoning).
* **Как работает атака:** Злоумышленник генерирует адрес, который визуально похож на один из адресов, с которыми пользователь часто взаимодействует (совпадают первые и последние символы), и отправляет с него `0 TON`. Адрес мошенника попадает в историю транзакций, и невнимательный пользователь может случайно скопировать его в будущем.
* **Как работает защита:** При вводе адреса на экране отправки (`WalletSend.vue`), `Address Guard` выполняет следующие проверки:
  1. Если адрес находится в `whitelist` (белый список успешных переводов пользователя), риск оценивается как `Low` (Безопасный).
  2. Если адрес новый и не встречается в истории, риск оценивается как `Medium` (Средний), предупреждая пользователя быть внимательнее.
  3. Если адрес **не** в `whitelist`, но при этом **очень похож** (совпадают начало и конец) на один из адресов в истории транзакций, система бьет тревогу и выставляет `Critical` риск. Пользователь получает ярко-красное предупреждение и окно дополнительного подтверждения, чтобы предотвратить отправку средств мошеннику.
